# Contextual Messaging & Meeting Notes Integration

## Problem

Messages generated by Revive.ai are not contextual enough. The AI doesn't have sufficient context from GHL conversations or call transcripts to write messages that feel like they came from someone who was on the call. Additionally, email formatting doesn't visually differ from SMS in the UI.

## Solution: Deep Context Pipeline (Approach A)

### Architecture

When "Generate Message" is clicked, the backend gathers context from 3 sources in parallel, merges them, and feeds a rich prompt to Claude.

```
Click "Generate" for Deal X
    |
    |-- [parallel] GHL: ALL conversation threads + contact metadata
    |-- [parallel] Fireflies OR Fathom: transcripts by contact email
    |
    v
Merge into unified context -> Claude prompt (channel-aware) -> Message
```

### Components

#### 1. Meeting Notes Service (Backend)

Pluggable service pattern. One active at a time based on which API key the user has connected.

- `MeetingNotesService` base pattern with `get_transcripts_for_contact(email) -> List[MeetingContext]`
- `FirefliesService(api_key)` - GraphQL API, search by participant email
- `FathomService(api_key)` - REST API, same output format
- Factory: check user's keys -> return correct service or None
- `/tmp` cache with 10-min TTL per contact email

`MeetingContext` common format:
```python
{
    "title": str,
    "date": str,
    "summary": str,
    "action_items": List[str],
    "keywords": List[str],
    "key_topics": List[str],
    "highlights": List[str],  # notable quotes or moments
    "participants": List[str]
}
```

#### 2. Enhanced GHL Context

Expand `get_deal_conversations()`:
- Fetch ALL conversation threads (remove cap of 2)
- Pull contact custom fields via `/contacts/{id}` endpoint
- Pull contact notes if available
- Include tags and pipeline stage in context

#### 3. Integrations Settings UI (Frontend)

New "Integrations" section in settings page:
- Fireflies card: API key input, Connect button, status badge
- Fathom card: API key input, Connect button, status badge
- Connecting one auto-disconnects the other (mutually exclusive)
- Test Connection button verifies key works
- API keys stored on user model: `fireflies_api_key`, `fathom_api_key`

#### 4. Email vs SMS Display Fix (Frontend)

- Email mode: subject line header, paragraph spacing, professional card layout
- SMS mode: current bubble-style display (unchanged)
- Channel selector triggers re-generation in correct format
- Visual distinction between email and SMS previews

#### 5. User Model Changes (Backend)

Add to User model:
- `fireflies_api_key: Optional[str]`
- `fathom_api_key: Optional[str]`

New API endpoints:
- `PUT /api/v1/settings/integrations` - save/remove integration keys
- `POST /api/v1/settings/integrations/test` - test connection

### Data Flow

1. User connects Fireflies or Fathom in Settings (stores API key)
2. User clicks "Generate Message" on a stalled deal
3. Backend fetches GHL conversations + meeting transcripts in parallel
4. All context merged into unified format
5. Claude receives rich prompt with full conversation + call transcript context
6. Returns channel-appropriate message (SMS sequence or email with paragraphs)
7. Frontend renders with channel-specific styling

### Key Decisions

- Either Fireflies or Fathom, not both simultaneously
- Live API calls at generation time (not background sync)
- `/tmp` caching prevents repeated API calls on regenerate
- Static `fireflies_cache.json` removed
- Channel switch triggers re-generation (not just display change)
